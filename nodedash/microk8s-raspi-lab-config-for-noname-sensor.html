Unlocking MicroK8s for Security Sensors

MicroK8s on Ubuntu/Raspberry Pi is locked down by default. These steps open the cluster for eBPF sensors, security agents, and host-inspection containers.

Why this is needed: Your Intel cluster already had privileged pods allowed, /proc without hidepid, and permissive AppArmor — so the Helm chart just worked. MicroK8s on Ubuntu Pi ships with stricter defaults that block all of those things.
1
Disable Pod Security Admission Restrictions Most Important
Newer Kubernetes enforces Pod Security Admission by default. Label your namespaces as privileged.
Default namespace 
kubectl label ns default \
  pod-security.kubernetes.io/enforce=privileged \
  pod-security.kubernetes.io/audit=privileged \
  pod-security.kubernetes.io/warn=privileged --overwrite
noname-security namespace (if deploying there) 
kubectl label ns noname-security \
  pod-security.kubernetes.io/enforce=privileged \
  pod-security.kubernetes.io/audit=privileged \
  pod-security.kubernetes.io/warn=privileged --overwrite
Apply to all common namespaces at once 
for ns in kube-system default ingress metallb-system noname-security; do
  kubectl label ns $ns \
    pod-security.kubernetes.io/enforce=privileged \
    --overwrite
done
2
Disable AppArmor Security
Ubuntu + MicroK8s often blocks access to /proc/.../ns/net via AppArmor profiles. Disable for lab use.
Disable the service 
sudo systemctl stop apparmor
sudo systemctl disable apparmor
Verify it's inactive 
sudo aa-status
Expected output: apparmor module is loaded but profiles should show as disabled or inactive.
3
Remount /proc — Remove hidepid Common Root Cause
This is the most common cause of "permission denied" errors in security sensors. If /proc is mounted with hidepid=2, containers can't see host processes.
Check current mount options 
mount | grep " on /proc "
If you see hidepid=2 — fix it immediately 
sudo mount -o remount,hidepid=0 /proc
Make permanent (lab only) — edit /etc/fstab 
proc  /proc  proc  defaults  0  0
4
Verify Privileged Containers Work Verify
Quick smoke test — if this pod launches successfully, Kubernetes is no longer blocking privileged containers.
kubectl run test --rm -it \
  --image=ubuntu \
  --privileged \
  -- bash
5
Make kubectl Work Globally MicroK8s Quirk
MicroK8s uses microk8s.kubectl instead of the standard kubectl. Scripts and Helm hooks often expect the standard binary.
sudo ln -s /snap/bin/microk8s.kubectl /usr/local/bin/kubectl
6
Enable Required MicroK8s Add-ons Networking
These add-ons relax networking and storage restrictions that commonly block sensor and security tool deployment.
microk8s enable dns
microk8s enable hostpath-storage
microk8s enable ingress
microk8s enable metrics-server
7
Re-deploy Your Helm Chart Deploy
Clean uninstall first to ensure all previous state is cleared before reinstalling with the relaxed config.
helm uninstall noname-security -n noname-security
helm install noname-security ./helm \
  --namespace noname-security \
  --create-namespace
8
Verify Sensors Can See Host Processes Verify
Exec into your sensor pod and confirm it can access host network namespaces. This is the final proof that the permission problem is resolved.
Shell into the sensor pod 
kubectl exec -n noname-security -it <sensor-pod> -- sh
Test host namespace visibility 
ls /proc/1/ns/net
ls /proc/$(pgrep kubelet)/ns/net
If both commands return results (no permission errors), the fix is confirmed and sensors have full host process visibility.

Setting Intel Cluster MicroK8s Pi (default)
